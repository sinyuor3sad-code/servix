## 📌 ملخص تنفيذي

بناء ثلاث ميزات مترابطة داخل SERVIX:

1. **منيو ذكي (Smart Menu)** — صفحة عامة تعرض خدمات الصالون، تسمح للزبونة باختيار خدمات وتوليد رقم طلب تعطيه للكاشيرة

## 🔀 المساران الأساسيان

### المسار A: الكاشيرة تختار الخدمات (المباشر)

```
الزبونة عند الكاشير
    → الكاشيرة تختار الخدمات يدوياً في POS
    → الكاشيرة تدفع
    → يظهر QR على شاشة الكاشير
    → الزبونة تمسح QR
    → تفتح صفحة الفاتورة العامة
    → تقييم (اختياري)
```

### المسار B: الزبونة تختار الخدمات بنفسها (المنيو الذكي)

```
QR ثابت عند المدخل أو الكاشير
    → الزبونة تمسح QR
    → تفتح صفحة المنيو (استعراض + اختيار)
    → تختار الخدمات
    → تضغط "إنشاء الطلب"
    → يظهر رقم الطلب (مثل A482)
    → تعطي الرقم للكاشيرة
    → الكاشيرة تدخل الرقم في POS
    → يظهر الطلب مع الخدمات والسعر
    → الكاشيرة تقدر تعدّل (إضافة/حذف خدمة)
    → الدفع
    → صفحة الزبونة تتحول تلقائياً من رقم الطلب إلى الفاتورة (WebSocket)
    → تقييم (اختياري)
```

**القاعدة:** كلا المسارين ينتهيان بنفس كيان الفاتورة (Invoice) الموجود. الطلب الذاتي هو مجرد "prefill" للفاتورة، وليس نظام فواتير منفصل.

## 📱 صفحة المنيو الذكي (Smart Menu)

### الوظيفة
صفحة عامة بدون تسجيل دخول تعرض خدمات الصالون بأسعارها. الزبونة تستطيع:
- **الاستعراض فقط** — تشوف الخدمات والأسعار وتطلع
- **الاختيار والطلب** — تختار خدمات → يظهر زر "إنشاء الطلب" → يطلع رقم

### الرابط
```
{salon-slug}.servix.com/order
```

### اللغة
- تدعم العربية والإنجليزية (الخدمات عندها `nameAr` + `nameEn` في الـ schema)
- الزبونة تختار اللغة من الصفحة

### المحتوى
- شعار الصالون
- اسم الصالون
- رسالة ترحيبية (اختيارية)
- قائمة الخدمات مقسمة حسب الفئات (ServiceCategory)
- كل خدمة تعرض: الاسم + السعر + المدة
- **لا تعرض:** اسم الموظفة، حقول الحجز، التاريخ، الأوقات
- زر "إنشاء الطلب" يظهر فقط عند اختيار خدمة واحدة على الأقل

### الثيمات — 4 أشكال

| # | الاسم | الوصف | يناسب |
|---|-------|-------|-------|
| 1 | **Classic** | قائمة عمودية بسيطة، كل خدمة سطر فيه الاسم والسعر | صالونات راقية تحب البساطة |
| 2 | **Cards** | بطاقات مربعة/مستطيلة، شبكة عمودين | صالونات تحب العرض البصري |
| 3 | **Compact** | فئات تنفتح وتنغلق (accordion) | صالونات كبيرة فيها 30+ خدمة |
| 4 | **Elegant** | غلاف كبير أعلى + مسافات واسعة + تصميم فاخر | صالونات VIP |

**الأشكال الأربعة تُبنى من البداية.**

### الألوان — 8 لوحات مضبوطة

| # | الاسم | الإحساس |
|---|-------|---------|
| 1 | بنفسجي ملكي | فخامة، VIP |
| 2 | ذهبي دافئ | كلاسيكي، راقي |
| 3 | وردي ناعم | أنثوي، لطيف |
| 4 | أسود أنيق | عصري، جريء |
| 5 | أزرق هادئ | نظيف، مريح |
| 6 | أخضر طبيعي | طبيعي، عضوي |
| 7 | بني دافئ | دافئ، مريح |
| 8 | فوشيا جريء | حيوي، شبابي |

**كل لوحة = مجموعة ألوان متناسقة** (أزرار، خلفية، نصوص، حدود) وليست لون واحد. يتم تطبيقها عبر CSS variables.

**لا يوجد color picker حر.** الصالون يختار من هذه الثمانية فقط.

### "Powered by SERVIX"
- **الباقة الأساسية:** يظهر إجبارياً أسفل الصفحة — سطر صغير "Powered by SERVIX"
- **الباقة المتقدمة:** الصالون يستطيع إخفاءه

## 🔢 نظام أرقام الطلبات

### الصيغة
**حرف إنجليزي + 3 أرقام**: `A482`

### قواعد الحرف
- الحرف يتغير كل شهر:
  - يناير = A، فبراير = B، مارس = C ... ديسمبر = L
- بعد 12 شهر يبدأ من A مرة أخرى
- هذا يمنع التصادم بين طلب قديم وجديد

### قواعد الرقم
- يبدأ من 001 كل شهر
- يزيد تسلسلياً: A001, A002, A003...
- الحد الأقصى 999 طلب في الشهر لكل صالون (كافي جداً)

### انتهاء الصلاحية
- الطلب ينتهي بعد **30 دقيقة** إذا لم تدخله الكاشيرة
- بعد الانتهاء: صفحة الزبونة تعرض "انتهت صلاحية الطلب" + زر "إنشاء طلب جديد"

### حالات الطلب
```
pending   → الطلب تم إنشاؤه، بانتظار الكاشيرة
claimed   → الكاشيرة فتحت الطلب
paid      → تم الدفع ← الطلب يرتبط بالفاتورة
expired   → انتهت الصلاحية بدون استخدام
```

### عزل الصالونات
كل صالون عنده قاعدة بيانات منفصلة (tenant isolation). الرقم `A482` في صالون ريماس مستقل تماماً عن `A482` في صالون لمسة. لا تصادم.

## 🧾 صفحة الفاتورة العامة

### الوصفة
صفحة عامة بدون تسجيل دخول تعرض فاتورة مدفوعة.

### الرابط
```
{salon-slug}.servix.com/invoice/{public_token}
```

**`public_token`** = سلسلة عشوائية طويلة (64 حرف) غير قابلة للتخمين. **لا يُستخدم رقم الفاتورة كرابط.**

### سلوك الرابط
- **دائم** — الرابط يبقى فاتورة دائماً بعد الدفع، لا يرجع لصفحة الخدمات
- الفاتورة فيها زر "استعراض خدماتنا" يفتح صفحة المنيو كصفحة جديدة

### المحتوى
يعرض:
- شعار الصالون
- اسم الصالون
- رقم الفاتورة
- تاريخ ووقت العملية
- قائمة الخدمات (الاسم + السعر)
- السعر قبل الضريبة
- الضريبة
- الإجمالي
- حالة الدفع: "مدفوعة ✓"
- زر "تقييم تجربتك"
- زر "الاتجاه إلى الموقع" (Google Maps)
- زر "استعراض خدماتنا" (يفتح المنيو)

لا يعرض:
- بيانات داخلية
- اسم الكاشيرة
- رقم جوال الزبونة
- أي بيانات حساسة

### الثيمات
نفس ثيم المنيو (نفس الشكل واللون اللي اختاره الصالون). تجربة متناسقة.

### الأمان
- `public_token` عشوائي 64 حرف — غير قابل للتخمين
- كل فاتورة لها token منفصل
- الصفحة read-only فقط
- يمكن للمديرة: تعطيل الرابط / إعادة إنشاء token جديد (القديم يصبح غير صالح)
- إذا الرابط معطّل: تظهر صفحة "هذا الرابط لم يعد صالحاً"

## ⭐ نظام التقييم

### التدفق
```
بعد عرض الفاتورة:
    → "كيف كانت تجربتك؟"
    → ⭐⭐⭐⭐⭐ (1-5)
    
    إذا 4 أو 5:
        → "يسعدنا! هل تحبين مشاركة تجربتك على Google؟"
        → [زر: افتحي Google Maps]
        → الزبونة تفتح Google وتكتب مراجعتها من حسابها بنفسها
    
    إذا 1 أو 2 أو 3:
        → "نعتذر عن التجربة. هل تحبين إضافة ملاحظة؟"
        → [حقل نص اختياري]
        → [زر: إرسال]
        → يروح للوحة الأدمن فقط
        → لا يظهر زر Google
```

### القواعد
- التقييم **اختياري 100%** — الزبونة تقدر تتجاوزه
- التقييم **مرة واحدة لكل فاتورة** — لو فتحت مرة ثانية تشوف تقييمها السابق
- التقييم **لا يرتبط بموظفة** في الإصدار الأول — على التجربة ككل
- **لا يتم نشر أي مراجعة على Google من داخل SERVIX** — الزبونة تفتح Google بنفسها

### لوحة الأدمن — صفحة Customer Feedback

تعرض:
- متوسط التقييم الإجمالي (مثل 4.3/5)
- عدد التقييمات هذا الشهر vs الشهر الماضي
- قائمة التقييمات مرتبة بالأحدث:

| الحقل | الوصف |
|-------|-------|
| التاريخ | تاريخ التقييم |
| عدد النجوم | 1-5 |
| التعليق | نص الملاحظة إن وجد |
| رقم الفاتورة | رابط للفاتورة |
| مصدر الطلب | كاشيرة مباشرة / طلب ذاتي |
| حالة Google | لم تُعرض / عُرضت / تم النقر |
| حالة المتابعة | جديد / تمت المراجعة / تم التواصل |

## 💻 تجربة الكاشيرة (POS)

### إضافة على POS الحالي

**1. خانة رقم الطلب:**
- حقل إدخال أعلى صفحة POS: "إدخال رقم الطلب"
- الكاشيرة تكتب `A482`
- النظام يجلب الخدمات والسعر
- تظهر الخدمات في الفاتورة — **قابلة للتعديل** (إضافة/حذف)
- الكاشيرة تكمّل الدفع

**(المسار اليدوي العادي يظل يعمل كما هو — لا شي يتغير)**

**2. شاشة ما بعد الدفع:**

بعد نجاح الدفع تظهر:
- ✅ "تم الدفع بنجاح"
- المبلغ النهائي
- **QR كبير وواضح** (يُولّد من الـ public token وقت العرض — لا يُحفظ كصورة)
- رسالة: "امسحي الكود لعرض فاتورتك"
- [زر: إعادة عرض QR]
- [زر: إغلاق / عملية جديدة]

**3. إخفاء QR:**
- QR **لا يختفي بمؤقت عدواني**
- يختفي عند:
  - بدء عملية جديدة
  - أو ضغط "إغلاق"

### ملاحظة مهمة للمنفذ

> ⚠️ ملف POS الحالي (`apps/dashboard/src/app/(dashboard)/pos/page.tsx`) حجمه **113KB** (~3000+ سطر).
> **يجب إعادة هيكلته وتقسيمه لمكونات صغيرة قبل إضافة أي ميزات جديدة.**
>
> التقسيم المقترح:
> ```
> pos/
> ├── page.tsx              ← الملف الرئيسي (صغير، ينسق بين المكونات)
> ├── components/
> │   ├── ClientSelector.tsx
> │   ├── ServicePicker.tsx
> │   ├── PaymentPanel.tsx
> │   ├── OrderInput.tsx       ← جديد: خانة رقم الطلب
> │   ├── InvoiceSummary.tsx
> │   └── QRSuccessModal.tsx   ← جديد: QR بعد الدفع
> ```
>
> **يجب اختبار كل الوظائف الحالية بعد التقسيم قبل إضافة ميزات جديدة.**

## 🔗 الربط اللحظي (Realtime)

### التقنية: WebSocket
- `EventsGateway` موجود ومشتغل في `apps/api/src/shared/events/events.gateway.ts`
- يدعم rooms حسب tenant و user

### المطلوب
إضافة room لصفحة الطلب:

```
عند فتح صفحة الطلب:
  → الزبونة تنضم لـ room: order:{orderId}

عند دفع الكاشيرة:
  → السيرفر يرسل event: order:paid مع بيانات الفاتورة
  → صفحة الزبونة تتحول تلقائياً من رقم الطلب إلى الفاتورة
```

### قاعدة مهمة
**WebSocket تحسين تجربة فقط، ليس اعتماد أساسي:**
- لو الاتصال انقطع، الزبونة تضغط refresh وتشوف الفاتورة (لأن الحالة محفوظة في قاعدة البيانات)
- لا يجب أن ينكسر أي شي لو WebSocket ما اشتغل

```prisma
model Invoice {
  // ... الحقول الحالية كما هي ...

  // ──── جديد: الوصول العام ────
  publicToken          String?   @unique @map("public_token") @db.VarChar(64)

### موديل جديد: SelfOrder
```prisma
enum SelfOrderStatus {
  pending
  claimed
  paid
  expired
}

model SelfOrder {
  id          String          @id @default(uuid()) @db.Uuid
  orderCode   String          @map("order_code") @db.VarChar(10)  // مثل A482
  status      SelfOrderStatus @default(pending)
  services    Json            @db.JsonB  // [{serviceId, nameAr, nameEn, price, duration}]
  totalEstimate Decimal       @map("total_estimate") @db.Decimal(10, 2)
  monthLetter String          @map("month_letter") @db.VarChar(1)  // A-L
  expiresAt   DateTime        @map("expires_at") @db.Timestamptz()
  claimedAt   DateTime?       @map("claimed_at") @db.Timestamptz()
  createdAt   DateTime        @default(now()) @map("created_at") @db.Timestamptz()

  invoice     Invoice?

  @@unique([orderCode, monthLetter])
  @@index([status, expiresAt])
  @@index([orderCode])
  @@map("self_orders")
}
```

### موديل جديد: InvoiceFeedback
```prisma
model InvoiceFeedback {

  @@map("invoice_feedbacks")

## 🌐 الـ API Endpoints

### Public (بدون auth — تحتاج Rate Limiting)

| Method | Path | الوظيفة |
|--------|------|---------|
| `GET` | `/public/:tenantSlug/menu` | جلب خدمات الصالون + معلومات الثيم |
| `POST` | `/public/:tenantSlug/order` | إنشاء طلب ذاتي → إرجاع رقم الطلب |
| `GET` | `/public/:tenantSlug/order/:code` | جلب حالة الطلب (للزبونة) |
| `GET` | `/public/:tenantSlug/invoice/:token` | جلب بيانات الفاتورة العامة |
| `POST` | `/public/:tenantSlug/invoice/:token/feedback` | إرسال تقييم |

### Private (تحتاج auth — للكاشيرة والمديرة)

| Method | Path | الوظيفة |
|--------|------|---------|
| `GET` | `/salon/orders/pending` | قائمة الطلبات النشطة |
| `GET` | `/salon/orders/:code` | جلب طلب برقمه |
| `POST` | `/salon/orders/:code/claim` | الكاشيرة تأخذ الطلب |
| `POST` | `/salon/invoices/:id/generate-token` | توليد public token |
| `DELETE` | `/salon/invoices/:id/revoke-token` | تعطيل الرابط العام |
| `POST` | `/salon/invoices/:id/regenerate-token` | إعادة إنشاء token (القديم يبطل) |
| `GET` | `/salon/feedback` | قائمة التقييمات (للأدمن) |
| `PATCH` | `/salon/feedback/:id/status` | تحديث حالة المتابعة |

## 🖥️ شاشة العميل الجانبية (Customer Display) — مرحلة مستقبلية

> **ملاحظة:** هذه ميزة منفصلة تُبنى بعد اكتمال الميزات الثلاث أعلاه.

شاشة إضافية موجهة للزبونة أمام الكاشير (مثل شاشات السوبرماركت). تعرض:
- الخدمات المختارة والأسعار أثناء إعداد الفاتورة
- رسالة "تم الدفع" بعد نجاح الدفع
- QR الفاتورة تلقائياً
- شعار الصالون ورسالة ترحيبية أثناء الخمول (Idle)

**تعمل عبر WebSocket** — تتحدث تلقائياً بدون تدخل الكاشيرة.

**شرط البناء:** يجب أن تكون ميزة إضافية بصرية فقط. لا تؤثر على عمل الكاشيرة لو كانت معطلة أو غير متصلة.
