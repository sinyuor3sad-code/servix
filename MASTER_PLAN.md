## A. الحكم التنفيذي

SERVIX منصة SaaS متعددة المستأجرين بنموذج قاعدة بيانات لكل مستأجر، مبنية بمكدس حديث (NestJS + Next.js 15 + React 19). البنية المعمارية سليمة ومتقدمة على معظم المشاريع في مرحلتها. لكن النظام يعاني من **فجوات تحقق كارثية** تجعله غير مؤهل للإنتاج الجاد.

**التقييم الإجمالي الحالي: 6.2/10**

| السؤال | الإجابة |
|---|---|
| جاهز لـ 5 عملاء تجريبيين؟ | نعم، مع مراقبة يدوية مكثفة |
| جاهز لأول عميل يدفع؟ | **لا** — سباق حجوزات متزامنة يسبب حجوزات مكررة، لا اختبارات، لا مراقبة |
| جاهز لـ 50 عميل؟ | **لا** — مجمع اتصالات 50 يتشبع، لا توسع أفقي، لا دفع إلكتروني |
| جاهز لـ 200+ عميل؟ | **لا** — يحتاج إعادة تصميم مجمع الاتصالات + توسع أفقي كامل |
| جاهز لتدقيق استثماري؟ | **لا** — 7 ملفات اختبار من 504 تُقصي فوراً |

**أخطر 5 مشاكل مكتشفة:**

1. `[مؤكد]` **سباق حجوزات متزامنة** — طلبان في نفس اللحظة يمرران فحص التعارض ويُنشئان حجزين لنفس الموعد. لا قفل على مستوى قاعدة البيانات. (`appointments.service.ts:124-145`)
2. `[مؤكد]` **إنشاء قاعدة المستأجر خارج المعاملة** — إذا فشل إنشاء DB بعد إنشاء المستخدم، المستخدم موجود لكن لا يستطيع الدخول. (`auth.service.ts:202-209`)
3. `[مؤكد]` **لا إيقاف رشيق (Graceful Shutdown)** — عند إيقاف الحاوية، الطلبات النشطة تُسقط فوراً والمعاملات تُقطع. (`main.ts` — لا معالج SIGTERM)
4. `[مؤكد]` **ترحيل بفقدان بيانات** — `prisma db push --accept-data-loss` يُنفذ على كل قواعد المستأجرين. تغيير مخطط قد يحذف أعمدة بصمت. (`tenant-database.service.ts:145`)
5. `[مؤكد]` **تغطية اختبارات < 2%** — 7 ملفات اختبار، صفر اختبارات واجهة، صفر E2E، صفر اختبارات حمل.

## B. مصفوفة التقييم التفصيلية

### B1. البنية المعمارية — 8.2/10

**تعريف 10/10:** مكونات مفصولة بعقود واضحة، حقن اعتماديات كامل، لا اقتران مباشر بين الوحدات، حدود سياق واضحة (DDD)، قابلية استبدال أي مكون دون تأثير جانبي.

**الفجوات:**
- `[مؤكد]` خدمة التدقيق (Audit) موجودة كـ dead code — مُعرّفة لكن لا تُستدعى من أي controller أو service. (`audit.service.ts`)
- `[مؤكد]` لا معرّفات ارتباط طلبات (Correlation IDs) — لا يمكن تتبع طلب عبر Guards → Middleware → Service → DB
- `[مؤكد]` بعض المتحكمات تُنسق الاستجابات يدوياً بدل ترك الـ interceptor. (`clients.controller.ts:46-51`)
- `[مؤكد]` حزمة `@servix/validations` المشتركة فارغة (`export {}`) — كان يُفترض أن تحوي مخططات Zod مشتركة

### B2. تعدد المستأجرين — 8.0/10

**تعريف 10/10:** عزل كامل على مستوى DB + شبكة + تخزين، اختبارات عزل مُؤتمتة، مراقبة لكل مستأجر، حدود استخدام قابلة للتكوين، لا تسرب بيانات ممكن حتى نظرياً.

**الفجوات:**
- `[مؤكد]` مجمع اتصالات بإخلاء FIFO وليس LRU — المستأجر الأكثر نشاطاً قد يُخلى أولاً (`tenant-client.factory.ts:62` يستخدم `keys().next().value`)
- `[مؤكد]` تعارض مجمع الاتصالات مع PgBouncer — 50 عميل Prisma × 3-10 اتصالات = 150-500 اتصال، لكن PgBouncer يسمح بـ 200 فقط (`docker-compose.prod.yml:58`)
- `[مؤكد]` ترحيل مخططات المستأجرين تسلسلي — 1000 مستأجر = ساعات (`tenant-database.service.ts:58` حلقة `for...of`)
- `[مؤكد]` `--accept-data-loss` يعني أي تغيير مخطط يحذف بيانات يُقبل تلقائياً (`tenant-database.service.ts:145`)
- `[مؤكد]` لا تتبع لإصدار المخطط لكل مستأجر — لا تعرف أي مستأجر على أي إصدار
- `[مؤكد]` لا حدود استخدام (rate limiting) لكل مستأجر — فقط لكل IP

## H. تيارات العمل التفصيلية

### H1. تيار الاختبارات

**الضعف الحالي:** 7 ملفات spec من 504 ملف مصدر. صفر اختبارات واجهة. صفر E2E. صفر حمل.

**الحالة المستهدفة:** 60% تغطية خلفية، 40% تغطية واجهة، E2E لكل تدفق حرج، اختبار حمل مُوثق.

**المهام بالتفصيل:** ← انظر القسم I

**معيار القبول:**
- `jest --coverage` يُظهر 60%+ للخلفية
- Playwright يمر على 8 تدفقات حرجة
- k6 يُوثق: max RPS، p95 latency، نقطة الانهيار
- CI يرفض الدمج إذا التغطية أقل من الحد

**ما يعني "مُنجز بشكل صحيح":** كل PR يمر عبر بوابة اختبارات. لا يُدمج كود بدون اختبار للمنطق الجديد. خط أساس التغطية لا ينخفض أبداً.

### H2. تيار المراقبة والرصد

**الضعف الحالي:** NestJS Logger افتراضي، Sentry للأخطاء فقط (10% عينة)، لا correlation IDs، لا metrics، لا dashboards، لا alerting.

**الحالة المستهدفة:** ← انظر القسم J

**المهام:**
1. إضافة Correlation ID middleware يُولد UUID ويُمرره في كل سجل
2. تكوين Winston: JSON output، مستويات (error/warn/info/debug)، correlation ID في كل سطر
3. إضافة تسجيل طلبات Nginx (`access_log` مع response time)
4. نشر Prometheus exporter لـ NestJS (request count, latency histogram, error rate)
5. إضافة مقاييس مخصصة: pool size, evictions, Redis hits/misses, BullMQ queue depth
6. نشر Grafana مع 3 لوحات: API overview, tenant health, background jobs
7. نشر Uptime Kuma لمراقبة endpoints
8. إعداد تنبيهات: error rate > 5%, p95 > 2s, pool exhaustion, job failures

**معيار القبول:** كل طلب قابل للتتبع من Nginx إلى DB response. تنبيه يصل خلال 5 دقائق من أي خلل.

### H3. تيار قابلية التوسع

**الضعف الحالي:** نسخة API واحدة، مجمع 50 اتصال، لا CDN، لا read replicas.

**الحالة المستهدفة:** ← انظر القسم K

### H4. تيار التقوية الأمنية

**الضعف الحالي:** Redis down = إبطال رموز يتوقف، لا CSRF، لا فحص أمني في CI، سجل تدقيق dead code.

**المهام:**
1. إضافة fallback لإبطال الرموز في DB عند سقوط Redis
2. تفعيل سجل التدقيق على كل عمليات الكتابة الحرجة
3. إضافة `npm audit --production` في CI مع فشل على ثغرات عالية
4. إضافة SAST scanner (Semgrep أو CodeQL)
5. توثيق سياسة إدارة الأسرار

**معيار القبول:** CI يمنع دمج كود بثغرات عالية. كل عملية كتابة مُسجلة في audit log.

### H5. تيار موثوقية قاعدة البيانات

**الضعف الحالي:** `--accept-data-loss`، لا تاريخ ترحيل، فهارس مفقودة، ترحيل تسلسلي.

**المهام:**
1. إنشاء baseline migration من المخطط الحالي
2. استبدال `prisma db push` بـ `prisma migrate deploy`
3. إضافة عمود `schema_version` في platform DB لكل مستأجر
4. إضافة الفهارس المفقودة الـ 7+
5. إصلاح cascade deletes الـ 3
6. تحويل ترحيل المستأجرين من تسلسلي إلى متوازي (batch 10)
7. إضافة pre-migration validation (اختبار على نسخة)

**معيار القبول:** كل تغيير مخطط عبر migration file. لا `--accept-data-loss`. كل مستأجر يُعرف إصدار مخططه.

### H6. تيار توسع WebSocket

**الضعف الحالي:** Socket.io بدون Redis adapter — نسخة واحدة فقط.

**المهام:**
1. تثبيت `@socket.io/redis-adapter`
2. تكوين الـ adapter مع Redis الموجود

### الإشارات المطلوبة ولماذا

| الإشارة | لماذا | التنفيذ |
|---|---|---|
| **Request rate** (req/s) | معرفة الحمل الحالي | Prometheus counter على كل طلب HTTP |

| **Redis hit/miss rate** | فعالية الـ cache | Prometheus counters في CacheService |
| **BullMQ queue depth** | اكتشاف تراكم وظائف | Prometheus gauge لكل queue |
| **BullMQ failed jobs** | إشعارات ضائعة | Prometheus counter + alert |
| **Active WebSocket connections** | حمل WebSocket | Prometheus gauge من EventsGateway |
| **DB query time p95** | بطء استعلامات | Prisma middleware logging |
| **Login failures/min** | محاولات اختراق | Prometheus counter |
| **Correlation ID** | تتبع طلب كامل | UUID في middleware، يمر في كل log |

### اللوحات المطلوبة (Grafana)

1. **API Overview:** request rate, error rate, latency percentiles, active connections
2. **Tenant Health:** pool size, evictions, per-tenant request rate, subscription status distribution
3. **Background Jobs:** queue depth per queue, processing rate, failure rate, retry rate
4. **Infrastructure:** CPU, memory, disk, Redis memory, PostgreSQL connections
